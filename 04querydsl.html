    Query DSL¶

    Term Queries: Búsqueda de texto estructurado.¶

    Si nos fijamos en el mapa del índice, vemos que los campos de texto pueden ser de tipo text o keyword. 

    GET productos/_mapping

    • Tipo keyword: El sistema no los analizará, tanto para la búsqueda como al indexar, nos servirán para realizar agregaciones, sumatorios, estadísticas….
    • Tipo text: El sistema los analizará como vimos anteriormente tanto al indexar como al buscar. Elasticsearch nos permite tener campos como ambos tipos.

    Vamos a ver como realizar búsquedas en los tipos keyword. Tenemos la documentación en

    https://www.elastic.co/guide/en/elasticsearch/reference/current/term-level-queries.html

    Veamos algunos casos

    Range query¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/query-dsl-range-query.html

    Nos permite encontrar documentos en un rango(numérico, fechas….)

    Veamos un ejemplo

    Productos que tienen el precio entre 10 y 20 

    GET /productos/_search
{
 "query": {
    "range": {
        "precio": {
        "gte": 10,
         "lte": 20
    }
  }
 }
}

    Paginación¶

    La consulta anterior nos devuelve los 10 primeros de un total de 1401. Podemos indicar desde qué elemento y cuántos queremos que nos traiga:

    Por ejemplo, desde la posición 10 y 30 documentos. 

    GET /productos/_search
{

   "from": 10, 
    "size": 30, 
 "query": {
    "range": {
        "precio": {
        "gte": 10,
         "lte": 20
    }
  }
 }
}

    Es de mucha utilidad para las búsquedas paginadas en un sitio web.

    Exists query¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/query-dsl-exists-query.html

    Como los documento de un índice no tienen que tener los mismo campos, podemos comprobar si tenemos un determinado campo 

    "query": { "exists": { "field": "description" } }

    Term query¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/query-dsl-term-query.html

    Nos permite buscar un término exacto en un campo keyword, como tenemos algunos campos con doble tipo(text/keyword), hay que utilizar el que identifica a keyword. Si vemos el mapping, el campo keyword para empresa sería 

    empresa.raw

    Para realizar una búsqueda exacta, sería 

    GET /productos/_search
{
  "query": {
    "term": {
      "empresa.raw": "BONAIRE"
    }
  }
}
    Nos da como resultado todos los productos de la empresa "BONAIRE".

    Si probamos con la consulta 

    GET /productos/_search
{
  "query": {
    "term": {
      "empresa.raw": "Bonaire"
    }
  }
}
    no nos devuelve resultados, ya que en la búsqueda no se realiza ningún análisis.

    Full text queries: Búsquedas para texto analizado¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/full-text-queries.html

    En estas búsqueda, la cadena de consulta se procesa utilizando el mismo analizador que se aplicó al campo durante la indexación y es donde radica todo el potencial de Elasticsearch.

    Match query¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/query-dsl-match-query.html

    Devuelve documentos que coincidan con un texto, un número, una fecha o un valor booleano proporcionados. El texto proporcionado se analiza antes.

    La consulta match es la consulta estándar para realizar una búsqueda de texto completo, incluidas las opciones para la coincidencia aproximada.

    Si probamos ahora a realizar la consulta anterior pero con match, vemos que el texto es analizado previamente(minúsculas..) para obtener el resultado 

    GET /productos/_search
{
  "query": {
    "match": {
      "empresa": "Bonaire"
    }
  }
}
    Si hacemos una consulta más compleja con varias palabras 
    GET /productos/_search
{
  "from": 1,
 "size": 30,
  "query": {
    "match": {
      "nombre": "Esmalte Jolie"
    }
  }
}

    Vemos que tenemos respuestas de las palabras con apariciones de ambos y separados. Por lo que deducimos que por defecto utiliza un “or” en la búsqueda. Si queremos cambiarlo para que aparezcan todos los términos. 

    GET /productos/_search
{
  "from": 1,
 "size": 30,
  "query": {
    "match": {
      "nombre": {
        "query":"Esmalte Jolie ",
         "operator":"and" 
      }
    }
  }
}
    También podemos ajustar para que al menos aparezcan un número determinado de términos

    "minimum_should_match": 2

    Análisis de la búsqueda¶

    Es de mucha utilidad conocer cómo analiza la búsqueda 

    GET productos/_analyze
{
  "text": ["limpiador de  muebles"],
  "analyzer": "mianalizador"
  , "explain": true
}

    Fuzziness¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/query-dsl-match-query.html#query-dsl-match-query-fuzziness

    En ocasiones, por ejemplo en la búsqueda en una web, en la que el usuario introduce el texto, es frecuente que se equivoque al introducir los datos:

    vebidas en vez de bebidas

    Podéis probar que no encuentra resultados con “vebidas”

    Mediante el parámetro Fuzziness, el sistema busca palabras cercanas 

    GET /productos/_search
{
    "query": {
        "match": {
            "nombre": {
                "query": "vebidas",
                 "fuzziness": "AUTO" 
            }
        }
    }
}

    March-phrase¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/query-dsl-match-query-phrase.html#query-dsl-match-query-phrase

    Si deseamos que coincida una frase completa 

    GET /productos/_search
{
    "query": {
        "match_phrase": {
            "nombre": {
                "query": "mata cucarachas aerosol"                
                }
        }
    }
}
    Ponemos indicar que falte alguna de las palabras mediante 
    GET /productos/_search
{
    "query": {
        "match_phrase": {
            "nombre": {
                "query": "mata  aerosol",
                 "slop": 1                
                }
        }
    }
}

    Multi_match¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/query-dsl-multi-match-query.html

    Podemos crear búsquedas en varios campos 

    GET /productos/_search
{
    "query": {
       "multi_match": {
           "query": "zapatos limpia",
           "fields": ["nombre","categoria"]
       }
    }
}
    y si necesitamos dar más peso a un campo, por ejemplo a categoría 
    GET /productos/_search
{
    "query": {
       "multi_match": {
           "query": "zapatos limpia",
           "fields": ["nombre","categoria^3"]
       }
    }
}

    Boolean Query¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/query-dsl-bool-query.html

    Permite combinar diferentes queries. Tenemos las siguientes opciones:

    must La query debe aparecer en documentos coincidentes y contribuirá a la puntuación. Es como un And
    filter La query debe aparecer en documentos coincidentes. Sin embargo, a diferencia de must, la puntuación se ignorará.
    should La Query debería aparecer en el documento correspondiente. Es como un Or. Permite subir el score a las coincidencias de must
    must_not La cláusula (consulta) no debe aparecer en los documentos coincidentes. Las cláusulas se ejecutan en el contexto del filtro, lo que significa que la puntuación se ignora y las cláusulas se consideran para el almacenamiento en caché. Como se ignora la puntuación 0, se devuelve una puntuación de para todos los documentos. Es como un Not

    Vamos a buscar los productos de “betún” pero que no sean de la empresa "bufalo". 

    GET productos/_search
{
    "query": {
        "bool": {
            "must": [
               { "match": {"nombre": "betun"}}
           ],
            "must_not": [
            {"match":{"empresa":"bufalo"}}
             ]
        }
    }
}
    Podemos añadir condiciones, por ejemplo, que el precio sea menos de 2 
    GET productos/_search
{
    "query": {
        "bool": {
            "must": [
               { "match": {"nombre": "betun"}},
               { "range": {"precio": {"lte": 2}}}
           ],
            "must_not": [
            {"match":{"empresa":"bufalo"}}
             ]
        }
    }
}
    Si queremos indicar condiciones de tipo OR. Por ejemplo: “Red bull” o “monster” 
    GET productos/_search
{
    "query": {
        "bool": {
            "should":  [
               { "match_phrase": {"nombre": "red bull"}},
               { "match": {"nombre": "monster"}}
           ]
        }
    }
}
    Puede ser necesario dar más importancia a alguna de las condiciones, le podemos asignar un peso extra, por ejemplo, queremos dar más importancia a “monster” porque está patrocinado por lo que queremos que aparezca en primer lugar estos productos: 
    GET productos/_search
{
    "query": {
        "bool": {
            "should":  [
               { "match_phrase": {"nombre": "red bull"}},
               { "match": {"nombre":{"query": "monster","boost":20}}}
           ]
        } 
    }
}
    La condición should tambien nos permite dar prioridad en el _score a una de las condiciones.

    La siguiente búsqueda, pedimos “chorizo o salchichón”, pero damos preferencia a la empresa “NOEL” 

    GET productos/_search?size=100
{
  "query": {
    "bool": {
      "must": [       
        {
          "match": {
            "nombre": "chorizo salchichon"
          }
        }
      ],
      "should": [
        {
          "term": {
            "empresa.raw": "NOEL"
          }
        }
      ]
    }
  }
}

    Filter¶

    No vamos a ver filter pero nos permite filtrar documentos pero sin asignar puntuación, por lo que la búsqueda es más rápida

    Geo queries¶

    Las búsquedas relacionadas con la geolocalización y distancia son muy interesantes para las búsquedas, sobre todo en las apps de móviles al estilo Wallapop o de tipo social.

    Agregaciones¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/search-aggregations.html

    https://medium.com/altoros/aggregations-en-elasticsearch-d2d70e9438ec

    Las agregaciones se pueden comparar con el Group By de SQL, y es fundamental para la analítica de datos. Elasticsearch maneja con enorme potencial las búsquedas agregadas y como vemos en el artículo anterior, nos permitirá crear búsquedas categorizadas al estilo Amazon, ya que nos devuelve tanto los productos como las agregaciones en el mismo resultado.

    Vamos a ver brevemente algunos ejemplos.

    El Total de productos por empresa. Es necesario utilizar un campo keyword

    sin documentos

    Si no queremos los documentos tenemos que indicarle size=0

    GET productos/_search
{
   "size": 0,

    "aggs": {
        "PorEmpresa":{
            "terms": {
                "field": "empresa.raw"
            }
        }
    }
}

    Se puede ver que es parecido a 

    SELECT empresa, count(*) FROM productosGROUP BY empresa ORDER BY count(*) DESC;
    Por otro lado, nos devuelve los 10 primeros, si queremos más, se lo indicaremos con size 
    GET productos/_search
{
   "size": 0,   
    "aggs": {
        "PorEmpresa":{
            "terms": {
                "field": "empresa.raw",
                 "size": 30 
            }
        }
    }
}
    ver las empresas que venden gaseosas 
    GET productos/_search
{

    "query": {
        "match": {"tipo":"gaseosas"}
    },
    "aggs": {
        "PorEmpresa":{
            "terms": {
                "field": "empresa.raw"
            }
        }
    }
}
    es como 
    SELECT empresa, count(*) FROM productos WHERE tipo=”gaseosa” GROUP BY empresa ORDER BY count(*) DESC;
    Vemos que nos devuelve la lista de documentos y al final la agregación

    Si no queremos los documentos tenemos que indicarle size=0 

    GET productos/_search
{
     "size": 0,
    "query": {
        "match": {"tipo":"gaseosas"}
    },
    "aggs": {
        "PorEmpresa":

…..

    Podemos ver el valor de venta máximo y mínimo 

    GET productos/_search
{
   "size": 0,
    "query": {
        "match": {"tipo":"gaseosas"}
    },
    "aggs": {
        "Maximo":{
           "max":{"field":"precio"}
        },
        "Minimo":{
           "min":{"field":"precio"}
        }
    }
}
    Es parecido en SQL a 
    SELECT max(precio),min(precio)  FROM productos WHERE tipo=”gaseosa”

    Cardinalidad¶

    Si necesitamos saber el número total de empresas diferentes tenemos el operador cardinality

    GET productos/_search?size=0
{
  "aggs": {
    "TotasEmpresa": {
      "cardinality": {
        "field": "empresa.raw"
      }
    }
  }
}

    Estadísticas¶

    Tenemos un operador que nos saca todos los datos estadísticos 

    GET productos/_search
{
   "size": 0,
    "query": {
        "match": {"tipo":"gaseosas"}
    },
    "aggs": {
        "Estadisticas":{
           "stats":{"field":"precio"}
        }
    }
}

    más estadísticas

    Si utilizamos extended_stats tenemos unos datos estadísticos más completos. Podéis probarlo.

    Nos puede interesar para mostrar algún tipo de nube de aparición de términos de nuestra índice

    Como hemos visto, no se puede realizar agregación de un campo analizado, 

    GET productos/_search
{
   "size": 0,

    "aggs": {
        "Categoria":{
            "terms": {
                "field": "nombre"
            }
        }
    }
}
    nos muestra un error Si nos fijamos en el detalle del error, nos recomienda definir el campo como

    Esto tiene un gran coste para el índice, pero nos permitiría realizar el conteo de palabras.

    No lo vamos a hacer ya que tendríamos que reconstruir el índice pero si lo necesitárais, tendrías que cambiar la propiedad del campo

    Histogram aggregation¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/search-aggregations-bucket-histogram-aggregation.html

    Nos puede interesar realizar agregaciones por intervalos, por ejemplo, contar el número de productos por intervalos de precio de 3€ 

    POST /productos/_search?size=0
{
  "aggs": {
    "Precios": {
      "histogram": {
        "field": "precio",
        "interval": 3
      }
    }
  }
}

    Más histogramas

    Podemos definir los intervalos nosotros. Podéis ver ejemplos en

    https://www.elastic.co/guide/en/elasticsearch/reference/current/search-aggregations-bucket-range-aggregation.html

    Muy interesantes son los intervalos por fechas

    https://www.elastic.co/guide/en/elasticsearch/reference/current/search-aggregations-bucket-datehistogram-aggregation.html

    y las de geoposicionamiento y distancia por intervalos

    https://www.elastic.co/guide/en/elasticsearch/reference/current/search-aggregations-bucket-geodistance-aggregation.html

    Agregaciones anidadas:¶

    Si necesitamos realizar una consulta al estilo SQL en el que queremos saber el total de productos por empresa y tipo 

    SELECT empresa, count(*), tipo, count(*)  FROM productos WHERE tipo=”gaseosa” GROUP BY empresa, tipo
    Tenemos que anidar agregaciones: 
    GET productos/_search?size=0
{
    "aggs": {
        "PorEmpresa":{
            "terms": {
                "field": "empresa.raw"
            },
            "aggs":{
               "PorTipo":{
                    "terms":{
                        "field":"tipo.raw"
                    }
                }
            }
        }
    }
}
    Si además queremos el precio máximo y mínimo 
    GET productos/_search?size=0
{
    "aggs": {
        "PorEmpresa":{
            "terms": {
                "field": "empresa.raw"
            },
            "aggs":{
                "PorTipo":{
                    "terms":{"field":"tipo.raw"},
                    "aggs": {
                        "Maximo":{"max":{"field":"precio"}},
                        "Minimo":{"min":{"field":"precio"}}
                    }
                }
            }
        }
    }
    }
}

    Agregaciones combinadas¶

    Podemos realizar varias peticiones de agregación en la misma consulta. Por ejemplo, si queremos saber los productos con “chocolate”:

    • los totales por empresa
    • las estadísticas
    • saber el total por rango de precios 
      GET productos/_search?size=0
{
  "query": {
    "match": {
      "nombre": "chocolate"
    }
  },
  "aggs": {
    "PorEmpresa": {
      "terms": {
        "field": "empresa.raw",
        "size": 10
      }
    },
    "Estadisticas":{
      "stats": {
        "field": "precio"
      }
    },
    "PorPrecio":{
      "filters": {
        "filters": {
          "baratos": {
            "range": {
              "precio": {
                "lt": 2
              }
            }
          },
          "medio": {
            "range": {
              "precio": {
                "gte": 2,
                "lte": 5
              }
            }
          },
          "caros":{
            "range": {
              "precio": {
                "gt": 5
              }
            }
          }
        }
      }
    }
  }
}

    Elasticsearch SQL¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/sql-getting-started.html

    Desde las últimas versiones de Elastic, podemos realizar consultas SQL. No está definido todo el estandar sql pero nos puede facilitar las consultas en Elasticsearch si no conocemos el lenguaje DSL del sistema

    POST _sql
{
  "query": """
  SELECT * FROM "productos" WHERE "empresa"='CALIFORNIA' 
  """  
}
    Podemos cambiar el formato en el que se nos devuelven los datos, por ejemplo, en vez de json podemos obtener texto en forma de tabla

    https://www.elastic.co/guide/en/elasticsearch/reference/current/sql-rest-format.html 

    POST _sql?format=txt
{
  "query": """
  SELECT * FROM "productos" WHERE "empresa"='CALIFORNIA'
  """
  }

    Podéis consultar la ayuda oficial para más información.

    Transformar sentencia SQL a lenguaje query DSL de Elasticsearch¶

    Elasticsearch lo que entiende es el lenguaje DSL, por lo que tiene que transformar la sentencia SQL. Podemos obtener el resultado de esa transformación para analizar si es lo que necesitamos o utilizarla como query definitiva que siempre será más eficaz. 

    POST _sql/translate
{
  "query": """
  SELECT "nombre" FROM "productos" WHERE "empresa"='CALIFORNIA'
  """  
}

    Ejercicio 5: Query DSL¶

    2. Busca los productos que tengan menor o igual precio que 20. Que te muestre 50 productos

    4. Realiza la misma búsqueda pero con "match".
    5. Qué diferencia tenemos entre una y otra búsqueda.
    6. Busca los productos donde aparezca "jamon" o "ahumado"
    7. Busca los productos que aparezcan tanto "jamon" como "ahumado". Recuerda el operador "and" de "match"
    8. Busca el texto "Los vinos de Navarra"
    9. Crea un sentencia que analice con "mianalizador" el texto "Los vinos de Navarra" y observa el resultado 10.Crea una sentencia para la frase :"los Binos de Nabarra" Observa el resultado.
    10. Ahora aplica fuzziness y observa el resultado 
      GET /productos/_search
{

  "query": {
    "match": {
      "nombre": {
        "query":"los Binos de Nabarra",
        "fuzziness": "AUTO"
      }
    }
  }
}
      qué ha conseguido el analizador.

    Boolean Queries

    1. Busca los vinos argentino o rioja: Puedes crear una sentencia "bool" donde obligas(must) a que sea "vino" y otra que sea "argentino rioja"
    2. Añade una condición para que el precio sea mayor de 20
    3. La misma que anteriormente pero queremos que aparezcan primero los "rioja". Recuerda que mediante "should" le damos más _score a la condición de búsqueda.

    Agregados

    1. Muestra mediante agregados las empresas que venden vino y junto al total de vinos que venden
    2. Saca las estadísticas de los precios de los productos donde aparece vino. Muestra las estadísticas de los precios de los vinos por empresa. Recuerda que tendrás que crear una agregación dentro de otra
