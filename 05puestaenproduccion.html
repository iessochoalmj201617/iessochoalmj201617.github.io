    Ejemplo de creación de query DSL y los retos que nos plantea¶

    Vamos a estudiar un posible caso de uso simplificado en la que se nos pide que creemos una query para ElasticSearch para una tienda en la que el objetivo es mostrar el mejor resultado al usuario de su búsqueda. Abre el fichero 05-consultas.txt”

    1. Crear los cinco productos de la tienda

    2. Imaginamos que el usuario el usuario realiza la búsqueda

      “tenis de uso diario”

      Nota: El sentido de tenis aquí es de zapatilla que es utilizado en algunas regiones

    3. La primera consulta que probamos es 

      GET  tienda/_search
    {
      "query": {
        "match": {
          "descripción": "tenis de uso diario"
        }
      }
    }

    Recuerda

    Recordaros que por defecto, las palabras que aparecen es como un or lógico de aparición de cada una de ellas y que Elastic ordena según relevancia de aparición, como “uso” aparece mucho en el documento 2, lo muestra el primero. Por otro lado, el documento 4 que es nuestro objetivo aparece el último

    • Vamos a intentar mejorar pidiendo que busque aquellos documento con todas las palabras mediante el operador and 
              GET  tienda/_search
        {
          "query": {
            "match": {
              "descripción": {
                "query": "tenis de uso diario",
                "operator": "and"
              }
            }
          }
        }
      Vemos que solamente nos devuelve uno que es un armario. Es demasiado restrictivo

    • Vamos a probar a indicarle que utilice un or, pero, que al menos aparezcan 2 palabras. 

      GET  tienda/_search
{
  "query": {
    "match": {
      "descripción": {
        "query": "tenis de uso diario",
        "operator": "or",
        "minimum_should_match": 2
      }
    }
  }
}
      Vemos que seguimos igual y si probamos con 3 ni siquiera aparece el resultado deseado.

    • Es necesario alterar el score de la búsqueda para afinar el resultado y que muestre arriba los resultados de interés al usuario. Para ello, añadiremos el operador should sobre el campo “nombre” para que si alguna de las palabra aparece, le suba el score al documento, aunque no sea obligatorio. 

          GET  tienda/_search
    {
      "query": {
        "bool": {
          "must": [
            {
              "match": {
                "descripción": "tenis de uso diario"
              }
            }
          ],
          "should": [
            {
              "match": {
                "nombre": "tenis de uso diario"
              }
            }
          ]
        }
      }
    }
      Vemos que ahora sí que aparece el documento deseado arriba, pero también nos aparece la pintura. Vamos intentar mejorar añadiendo restricciones y modificando el score mediante el parámetro boost que permite dar peso a ciertas condiciones

        GET  tienda/_search
    {
      "query": {
        "bool": {
          "must": [
            {
              "match": {
                "descripción": "tenis de uso diario"
              }
            }
          ],
          "should": [
            {
              "match_phrase": {
                "nombre": {
                  "query": "tenis de uso diario",
                  "boost": 4
                }
              }
            },
            {
              "match": {
                "nombre": {
                  "query": "tenis de uso diario",
                  "boost": 3
                }
              }
            },
            {
              "match_phrase": {
                "descripción": {
                  "query": "tenis de uso diario",
                  "boost": 4
                }
              }
            }
          ]
        }
      }
    }

    Conclusión¶

    Conclusión

    Vemos que es un proceso complejo en el que es necesario la optimización y la mejora constante observando siempre si el funcionamiento es el correcto. No hemos metido en el proceso los analizadores y sus mejoras de idiomas, así como la posibilidad de trabajar con sinónimos, pero nos podemos hacer una idea de la necesidad de un estudio detallado de las necesidades del proyecto y realizar diversas pruebas de tipo A/B para mejorar la búsqueda

    Plantillas de búsqueda¶

    https://www.elastic.co/guide/en/elasticsearch/reference/current/search-template.html

    Una vez tenemos la consulta que necesitamos, tenemos Elasticsearch dentro de nuestro backend, al equipo desarrollador web no debemos entregarle la api de la consulta para que realice las llamadas a ElasticSearch, lo que hacemos es crear una plantilla parametrizada con una interface de entrada que utilizará y que al equipo de desarrollo de Elasticsearch le permita ir depurando la consulta sin que afecte al proyecto web.

    Una plantilla de búsqueda es una búsqueda almacenada que se puede ejecutar con diferentes variables. Para el ejemplo anterior del índice de la “tienda”, para simplificar el código, utilizaremos una de las primeras soluciones:

    Creamos la plantilla¶

    PUT _scripts/busqueda_tienda
{
  "script": {
    "lang": "mustache",
    "source": {
       "query": {
    "bool": {
      "must": [
        {
          "match": {
            "descripción": "{{parametro}}"
          }
        }
      ],
      "should": [
        {
          "match": {
            "nombre": "{{parametro}}"
          }
        }
      ]
    }
  }
    }
  }
}
    Podemos testear la plantilla viendo la consulta que genera 
    POST _render/template
{
  "id": "busqueda_tienda",
  "params": { 
    "parametro":"tenis de mesa"
    } 
}

    Lanzamos la plantilla contra el índice¶

    GET tienda/_search/template
{
 "id": "busqueda_tienda",
  "params": { "parametro":"tenis de mesa"}
}

    Ejemplo propuesto en el curso de iniciación de Elasticsearch¶

    Una de las conferencias de la web de Elasticsearch propone un ejemplo muy interesante de un buscador sobre el blog de consultas de la propia web.

    https://www.youtube.com/watch?v=IOuGbkVBdo0&list=PL_mJOmq4zsHaWvXgiNj9-OsDhjBSfIitW

    La idea es crear una plantilla de búsqueda que será utilizada por una página web sin que esta sepa el contenido y complejidad de la consulta.

    En un proyecto real, el servidor Elasticsearch permanecerá oculto detrás del servidor de aplicaciones web.

    Por simplificación, vamos a utilizar una página html local que mediante tecnología AJAX consultará directamente mediante una petición REST a Elasticsearch.

    Modificación de la configuración de Elasticsearch¶

    Los navegadores están protegidos ante las peticiones AJAX a diferentes dominios al que pertenece la página. Vamos a tener que realizar una modificación en la configuración de Elasticsearch para que funcione en Chrome, pero no siempre funcionará en Firefox. Para más información

    https://developer.mozilla.org/es/docs/Web/HTTP/CORS

    • Cierra/Para el servidor Elasticsearch
    • Edita el fichero de configuración de la instalación de Elasticsearch (Un buen editor es Visual Studio code):

    config/elasticsearch.yml

    • Añade al final las líneas(cuidado con añadir espacios o tabulaciones indebidas, los ficheros yml son muy sensibles) 
      http.cors.enabled: true
http.cors.allow-origin: '*'
http.cors.allow-methods: "OPTIONS, HEAD, GET, POST, PUT, DELETE"
http.cors.allow-credentials: true
http.cors.allow-headers: "X-Requested-With, Content-Type, Content-Length,accept, authorization"

    • Inicia Elasticsearch
    • Inicia Kibana

    Creamos el índice blogs: Importación de blogs.csv¶

    Como hicimos con el productos.csv, vamos a importar una lista de artículos del blogs.csv de la web de Elasticsearch desde kibana

    Fichero blogs.csv

    • Nombra el índice como blogs. Pulsa importar.

    • Comprueba desde Dev tools de kibana que se ha creado 

      GET blogs/_search
{
  "query": {
    "match_all": {}
  }
}

    • Creamos la primera versión de la plantilla de búsqueda 

          PUT _scripts/busqueda_blogs
    {
      "script": {
        "lang": "mustache",
        "source": {
          "query": {
            "match": {
              "content": "{{terminos}}"
            }
          }
        }
      }
    }

    • Probamos que funciona 
      GET blogs/_search/template
        {
          "id": "busqueda_blogs",
          "params": {
            "terminos": "Logstash"
          }
        }

    • Descarga la página web: web_elastic.zip

    • Si editamos la página web busqueda.html, vemos en el código cómo se realiza la solicitud de búsqueda

    Cuando recibe de respuesta, añadimos la respuestas encontradas a la página web

    Si abrimos la página con Chrome, vemos como realiza las búsquedas sobre Elasticsearch

    Ejercicio opcional 3: Wordpress y Elasticsearch¶

    Actualmente, casi el 50% de sitios web están implementados mediante Wordpress. La instalación básica suele ser sobre una base de datos Mysql. Este servidor de bases de datos tiene la posibilidad de no ser lo suficiente en las siguiente situaciones:

    • Sitios donde la búsqueda es el principal medio de navegación.
    • Sitios WooCommerce con una gran cantidad de pedidos/productos donde los administradores del sitio deben poder buscar en la lista de pedidos regularmente.
    • Cualquier sitio con una gran cantidad de publicaciones donde las consultas MySQL producen resultados inaceptablemente lentos El siguiente artículo nos explica cómo podemos implantar Elasticsearch en nuestro sitio Wordpress utilizando el plugin ElasticPress, para potenciar la búsquedas.

    https://www.cloudways.com/blog/elasticsearch-on-wordpress/

    Realiza una instalación de Wordpress(puedes utilizar las versiones de bitnami para simplificar el proceso) y documenta el proceso de implantación de Elasticsearch como motor de búsqueda
